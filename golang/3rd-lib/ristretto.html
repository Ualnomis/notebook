1 概述

ristretto 是一个快速的并发缓存库，其重点是性能和正确性。

建立 Ristretto 的动机来自对 Dgraph 中无争用缓存的需求。

2 特性

  • 高命中率。结合独特的接受/逐出政策，Ristretto 的性能堪称一流。
    • 驱逐：SampledLFU–与精确的 LRU 相当，并且在搜索和数据库跟踪上具有更好的性能。
    • 接受：TinyLFU–额外的性能，几乎没有内存开销（每个计数器 12 位）。
  • 快速吞吐量。使用多种技术来管理竞争，进而获得出色的吞吐量。
  • 基于成本的逐出。任何被认为有价值的大型新 item 都可以驱逐多个较小的 item（成本可以是任何东西）。
  • 完全并发。可以使用任意数量的 goroutine，而不会降低吞吐量。
  • 统计。吞吐量、点击率和其他统计信息的可选性能统计。
  • 简单的 API。只需指定理想的 Config 值，即可开始运行。

3 使用

3.1 示例

func main() {
    cache, err := ristretto.NewCache(&ristretto.Config{
        NumCounters: 1e7,     // number of keys to track frequency of (10M).
        MaxCost:     1 << 30, // maximum cost of cache (1GB).
        BufferItems: 64,      // number of keys per Get buffer.
    })
    if err != nil {
        panic(err)
    }

    // set a value with a cost of 1
    cache.Set("key", "value", 1)

    // wait for value to pass through buffers
    time.Sleep(10 * time.Millisecond)

    value, found := cache.Get("key")
    if !found {
        panic("missing value")
    }
    fmt.Println(value)
    cache.Del("key")
}

更多示例参见 ristretto测试用例

3.2 Config

3.3 NumCounters int64

NumCounters 是保留用于接纳和逐出的 4 位访问计数器的数量。经测试，将其设置为满时希望保留在缓存中的条目数的 10 倍，可以获得很好的性能。

例如，如果希望每个条目的成本为 1，而 MaxCost 为 100，则将 NumCounters 设置为 1,000。或者，如果使用可变成本值，但预计缓存已满时将容纳约 10,000 个条目，请将 NumCounters 设置为 100,000。重要的是全部缓存中唯一条目的数量，不一定是 MaxCost 值。

3.4 MaxCost int64

MaxCost 决定逐出的方式。例如，如果 MaxCost 为 100，而成本为 1 的新条目将总缓存成本增加到 101，则将驱逐 1 个项目。

MaxCost 也可以用来表示最大大小（以字节为单位）。例如，如果 MaxCost 为 1,000,000（1MB），并且缓存中已包含 1,000 个 1KB 项，则新的（接受的）项将导致 5 个 1KB 的项被逐出。

MaxCost 可以是任何值，只要它与调用 Set 时使用成本值的方式匹配即可。

3.5 BufferItems int64

BufferItems 是 Get 缓冲区的大小。我们发现最佳价值是 64。

如果由于某种原因看到由于大量竞争而导致性能下降（异常情况），请尝试以 64 的增量增加此值。这是一种微调机制，可能不必了解。

3.6 Metrics bool

想要实时记录各种统计信息时，将该值设置为 true。之所以是 Config 标志，是因为吞吐量性能开销为 10％。

3.7 OnEvict func(hashes [2]uint64, value interface{}, cost int64)

每次驱逐都会调用 OnEvict。

3.8 KeyToHash func(key interface{}) [2]uint64

用于每个 key 的哈希算法。如果为 nil，则 Ristretto 根据底层接口类型具有多种默认值。

请注意，如果要使用 128 位哈希，则应使用完整的 [2]uint64，否则只需将 uint64 填充在 0 位置，它的行为就像任何 64 位哈希一样。

3.9 Cost func(value interface{}) int64

是一个可选功能，可以传递给 Config 以便在运行时评估项目成本，并且仅适用于未丢弃的 Set 调用（如果计算项目成本特别昂贵，而且条目最终会被丢弃，因为不值得浪费时间，则此功能非常有用时间）。

向 Ristretto 发出信号，表示想使用此 Cost 函数：

  1. 将 Cost 字段设置为非 nil 函数。
  2. 在为新项目或项目更新调用 Set 时，请使用成本 0。
