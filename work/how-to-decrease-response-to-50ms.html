1 效果

  1. 最复杂的接口响应从接手项目前的 1-2s，降低到现在的 100ms 以内。
  2. 数据库内存从高峰期的 13-14Gb，缩减到当前的 1Gb 以内。
  3. 代码模块化，可以复用，方便测试。
  4. 项目有完善的项目文档（markdown 格式）。
  5. 所有代码都有测试用例，有代码覆盖率展示。

2 业务代码

  • 接口处理流程规范化。
    • 所有处理流程规范为解析参数->请求处理->返回响应，使得处理逻辑更加清晰。
  • 设计 request 和 response 结构体，好处：
    • 替代原先滥用的 map[string]interface{} ，该结构使用发射进行类型转换，低效。
    • 结构清晰明了，可以和代码文档相对应，也便于日志打印。
  • 拆分处理函数，这么做有两个优点：
    • 便于代码测试。
    • 便于后面使用 goroutine，提高并发。
  • 原先通过 for 循环过滤大量数据库查询结果，现在将限制条件放置在 sql 查询语句中，并将结果缓存在包含查询条件的 key 中。
  • 整个处理过程只使用代表书籍的 coopID_bookID，只在需要返回书籍信息的时候的时候查询书籍详情。
  • 将数据库和缓存单独拆分，设计成组件方式，好处：
    • 方便替换别的数据库。
    • 方便测试数据库结构。
  • 集成 ELK 数据采集和钉钉群告警。
    • 数据可视化。
    • 及时处理线上问题。
  • 使用 zap 替代代低效的 fmt.Printf 写日志。
    • fmt.Printf 低效，没有日志分级。
    • 日志组件利用 zap 不仅打印高效，还可将运行日志和数据日志分离为不同的日志文件。
  • 程序本地缓存书籍信息。
    • 不用去 redis 查询，同时减少了从 redis 存取时候的 json 转换。

3 语言层面

  1. 将 beego 的 orm 替换成原生的 mysql。
  2. 使用协程提高请求处理流程速度。
  3. 使用 python 实现一些数据加载，脚本类型的功能。
  4. 使用 gocheck 添加代码测试用例。
  5. 使用 gin 替换 beego 框架。

4 缓存

  1. 使用合理的数据结构。
    • 使用 set 存储分类信息。
    • 使用 zset 存储排行信息。
  2. 设置合理的 key 名，将过滤条件包含在 key 中，方便查询。
  3. 只缓存合理的数据：
    • 静态配置数据和与动态数据数据分 DB 存储。
    • 只缓存书籍唯一标识（coopID_bookID）。
  4. 合理设置缓存过期策略。
    • 所有动态内容对应的 key 都设置有效期。
    • 数据库更新则删除缓存。

5 数据库（MySql）

  1. 合理设置设计表结构。
    • 拆分原来数据表中无关的数据字段。
    • bookCate 表与 bookInfo 表需要联合查询，优化时候就把 bookInfo 表中的信息插入到 bookCate 表中。
    • bookInfo 通过 online 字段进行拆分为不同的表，优点：
      • 可以通过外键方便的与其他表保证引用完整性，方便删除书籍。
  2. 合理的数据库字段类型。
    • 使用 set enum 等数据类型。
  3. 优化索引。
    • 调整 coop_bookID 索引位 bookID_coop。
    • 使用 fulltext 替换原有字符串正则表达式。
  4. 使用查询优化程序和慢查询日志查找问题和验证优化结果。

6 其他

  1. 代码仓库遵循 golang 规范构建，这样别的项目也能引用我们的代码。
  2. 使用 go mod 替代 vender。
  3. 给出支付服务解决用户订单并发处理的方法：
    • 序列化请求，然后使用 pod 对接请求序列。
    • 使用 redis 实现分布式锁，处理时对用户进行加锁。
    • 使用 nginx 将用户请求 hash(userid)到业务机器，业务机器串行处理请求，这种方式对现有代码改动最小。
